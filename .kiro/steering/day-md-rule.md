# TYPOvibe Day MD 파일 작성 규칙

## 적용 대상
각 day 폴더의 `dayN.md` 파일

## 필수 구조

```markdown
# Day N — 제목

**날짜 (요일)**

FROM · [출처명](URL)

## 출발
왜 이걸 만들게 됐는지. 어떤 글/영상/레퍼런스에서 영감을 받았는지. 2~4문장.

## 컨셉
무엇을 어떤 방식으로 표현하는지. 시각적 메타포가 있으면 설명. 핵심 아이디어.

## 구조
화면 구성, 레이어 구조, 인터랙션 흐름을 불릿으로 나열.

## 기술
### 주요 기술 영역 1
구체적인 구현 방법. 알고리즘, API, 라이브러리 사용법.
코드 스니펫이나 수식이 있으면 포함.

### 주요 기술 영역 2
...

## 삽질 기록
v1부터 최종 버전까지의 시행착오. 무엇을 시도했고, 왜 실패했고, 어떻게 해결했는지.

## 핵심 교훈
이 작업에서 배운 것. 다른 프로젝트에서도 쓸 수 있는 패턴이나 원칙.

## 파일 구조
```
public/works/dayN/
  파일 목록과 한 줄 설명
```

## 레퍼런스
영감을 준 작품, 사용한 도구, 참고한 문서 링크.

---

## 동료 공유용 요약

### 한 줄 요약
### 왜 이걸 만들었나
### 작업 흐름 (번호 리스트)
### 핵심 인터랙션 (불릿)
### 삽질 포인트 (불릿)
```

## 작성 원칙

1. **구체적으로** — "어려웠음" 이 아니라 "MIDI에서 멜로디 추출이 어려웠음. 피아노 편곡은 반주가 섞여있어서 최고음 추출, 음역대 필터링, stepwise motion 추적을 시도"
2. **삽질을 솔직하게** — 실패한 시도를 숨기지 말 것. v1→v2→...→최종 과정을 기록
3. **교훈은 재사용 가능하게** — "다음에 비슷한 걸 만들면 이렇게 하면 됨" 수준으로
4. **기술 섹션은 구현 수준으로** — 사용한 API 이름, 파라미터, 알고리즘 이름 명시
5. **파일 구조 포함** — 다른 사람이 코드를 찾아볼 수 있게
6. **동료 공유용 요약은 비개발자도 이해 가능하게** — 기술 용어 최소화, 결과물 중심

## 분량 기준
- day1, day11 수준 (약 150~300줄)
- 단순 요약이 아닌 **작업 일지 + 기술 문서** 역할
