# Confluence 업무 요약 워크플로우

## 페이지 생성 도구
- Confluence MCP(mcp-atlassian)를 사용하여 페이지 생성/수정/검색
- 본문은 Storage 형식(XHTML)으로 구성하여 `content_format: "storage"`로 전달
- wiki 마크업이나 markdown 포맷은 매크로가 깨질 수 있으므로 사용하지 않음
- 로컬에 wiki/json/content 파일을 생성하지 않고, 메모리에서 구성 후 MCP로 바로 전달

## 개인 스페이스 정보
- 스페이스 URL: `https://cloud.wiki.woowa.in/wiki/x/EQAzDQ`
- 실제 API URL: `https://woowahanbros.atlassian.net/wiki`
- 스페이스 키: `~mgmg92`
- 모든 페이지는 반드시 이 개인 스페이스에만 생성
- 다른 스페이스에는 절대 올리지 않음

## 페이지 계층 구조
```
개인 스페이스 (~mgmg92)
└── 데일리 바이브코딩 - 타이포그래피 (page_id: 1109624486)
    └── 연도 (yyyy)
        └── 월 (yyyy-mm)
            └── 일 페이지
```

## 페이지 제목 규칙
- 형식: `YYMMDD [카테고리] 제목`
- 예시:
  - `260416 [노래가사] 내가 바흐라니`
  - `260417 [뉴스칼럼] 로봇개가 개를 산책시켜..`
- 날짜는 6자리(YYMMDD), 카테고리는 대괄호로 감싸기
- 작업 내용은 세부 사항이 아닌 큰 개념으로 기재
- 프로젝트명은 현재 작업 중인 폴더명이나 사용자가 알려준 프로젝트명 사용

## 페이지 생성 절차 (필수)

1. 연도/월 추출: 제목에서 yyyy, yyyy-mm 추출
2. 연도 페이지 찾기: `confluence_get_page(title="yyyy", space_key="~mgmg92")`
3. 연도 페이지 없으면 생성: `confluence_create_page(space_key=..., title="yyyy", content="", parent_id="1042029562")`
4. yyyy-mm 페이지 찾기: `confluence_get_page(title="yyyy-mm", space_key=...)`
5. yyyy-mm 페이지 없으면 생성: `confluence_create_page(..., title="yyyy-mm", parent_id=연도페이지id)`
6. 최종 페이지 생성: `confluence_create_page(..., title=제목, parent_id=yyyy-mm페이지id, content=Storage XML, content_format="storage")`

## 사용자 확인 규칙
- wiki 페이지를 생성/업데이트하기 전에 반드시 사용자에게 구조와 내용을 먼저 보여주고 확인받기
- 이미지를 첨부할 수 있는 내용이면 사용자에게 먼저 확인
- 사용자가 이미지를 제공하겠다고 하면, 플레이스홀더를 넣고 페이지 생성 후 이미지 파일 경로를 받아 첨부

## 페이지 본문 구조 (Storage 형식)

### 1. 요약 표 (최상단)
- 본문 최상단에 h1 제목 없이 바로 요약 표부터 시작 (페이지 타이틀에 이미 제목이 있으므로)
- 항목 | 작업 내용 | 비고
- 항목 칸에 인덱스 넘버 넣지 않음
- 상태 열, 중요도 열 사용 안 함

### 2. 목차 + 핵심 요약 (two_equal 레이아웃)
- 요약 표 바로 아래 배치
- 왼쪽 셀: 목차 매크로
- 오른쪽 셀: 정보 패널(info) - 오늘의 모든 업무를 한 줄씩 통합 요약

### 3. 구분선 + 여백
- 목차+핵심요약 아래에 `<hr/>` 삽입
- 구분선과 첫 섹션 사이에 `<p><br/></p>` 한 줄 여백

### 4. 프로젝트별 상세 내용
- 섹션 타이틀: h2 + 이모지 (타이틀에만 이모지 사용, 본문에는 사용하지 않음)
- 기입 순서:
  1. 📋 진행 배경 (h2) - 왜 이 작업을 했는지
  2. ✨ 어떤 점이 좋아지는지 (h2) - 장점/변화/효과
  3. ⚙️ 어떻게 동작하는지 (h2) - 비전문가 눈높이로 흐름 설명
  4. 상세 내용 - Expand 매크로로 감싸기 (기술 구현 상세)

### 5. 간격 규칙 (중요)
- 같은 섹션 내 요소(타이틀 + 본문 + Expand)는 붙여서 하나로 묶기
- 섹션이 끝나는 지점에서만 `<p><br/></p>` 한 줄로 단락 구분
- 예시: "⚙️ 어떻게 동작하는지" + 본문 + Expand는 전부 붙어있어야 함

### 6. 막혔던 순간 섹션 (선택, 막힌 경험 있을 때 필수 포함)
- 타이틀: `<h2>⚠️ 막혔던 순간</h2>`
- 구조: warning 패널(빨간 박스) → 시도/해결(일반 텍스트) → info 패널(파란 박스)
- 원인: warning 패널 안에 배치
- 시도/해결: 패널 밖에서 일반 텍스트로 과정 서술
- 알게 된 점: info 패널 안에 배치 ("교훈" 대신 "알게 된 점" 사용)
- 각 항목 앞에 중앙점(·) 붙여서 구분
- Storage 포맷 예시:
  ```xml
  <h2>⚠️ 막혔던 순간</h2>
  <ac:structured-macro ac:name="warning" ac:schema-version="1">
    <ac:rich-text-body>
      <p><strong>문제 요약 한 줄</strong></p>
      <p>· 원인 설명</p>
    </ac:rich-text-body>
  </ac:structured-macro>
  <p><strong>· 시도</strong> - 시도한 내용</p>
  <p><strong>· 해결</strong> - 해결 방법</p>
  <ac:structured-macro ac:name="info" ac:schema-version="1">
    <ac:rich-text-body>
      <p><strong>알게 된 점</strong> - 배운 내용</p>
    </ac:rich-text-body>
  </ac:structured-macro>
  ```

### 7. 참고 정보
- 하이퍼링크: 사용자가 제공한 URL만 텍스트에 링크
- 코드/파일/경로: `<code>` 태그 사용

## 작성 스타일

### 말투
- "~했어요", "~이에요", "~돼요" 같은 부드러운 존댓말 사용
- 일상에서 말하듯 자연스럽게 쓰기
- 좋은 예: "반복해야 했어요", "만들고 싶었어요", "알아서 만들어져요", "해결했어요"
- 피할 것: "~거든요", "~더라고요" 같은 과한 구어체 / "과정이 반복적이었다" 같은 딱딱한 문어체 / 반말
- 결론/핵심 먼저, 이유를 뒤에
- 짧은 문장, 한 문장에 하나의 정보
- 기술 용어 사용 시 괄호 안에 한 줄 풀이

### 이모지/기호 규칙
- 이모지는 섹션 타이틀(h2)에만 사용, 본문/Expand 제목에는 사용하지 않음
- 기호는 1개만 사용 (여러 개 붙이지 않음)
- 구분 기호는 `-` 통일

### 디자이너 친화적 작성 규칙
- 읽는 사람은 개발자가 아니라 디자이너/기획자라고 가정
- 개발자 전용 개념(API, 함수명, 정규식 등)은 Expand 안에만
- 본문은 "무엇이 바뀌었는지", "어떤 문제가 해결되었는지" 중심
- 기술 용어 사용 시 괄호 안에 한 줄 풀이 필수
- 결과 중심 서술: "함수를 추가했다" 대신 "파일을 더 빠르게 찾을 수 있게 됐다"

### 표현 도구
| 용도 | Storage 포맷 |
|------|-------------|
| 코드/파일명/경로 | `<code>텍스트</code>` |
| 강조 | `<strong>` |
| 링크 | `<a href="...">텍스트</a>` (사용자가 준 URL만) |
| 코드 블록 | `ac:name="code"` 매크로 |
| 접기 | `ac:name="expand"` 매크로 |
| 정보 패널 | `ac:name="info"` 매크로 |
| 경고 패널 | `ac:name="warning"` 매크로 |

## 리서치 wiki 업데이트 규칙

### 개요
- 리서치 내용 갈무리 요청 시, 일자별 페이지가 아닌 리서치 페이지 하위에 작성
- 페이지 제목 형식: `[기술스택] 요약`
- 본문 작성 방법은 기존 업무 요약 워크플로우와 동일

### 페이지 계층 구조
```
개인 스페이스 (~mgmg92)
└── 리서치 (page_id: {RESEARCH_PAGE_ID})
    └── [기술스택] 요약 제목
```

### 페이지 생성 절차
1. 리서치 부모 페이지 확인
2. 중복 확인: 같은 기술스택 제목의 페이지가 이미 있는지 검색
3. 이미 있으면 업데이트, 없으면 새로 생성
